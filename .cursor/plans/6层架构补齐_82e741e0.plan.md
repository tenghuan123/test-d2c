# 6层架构缺口补齐计划

## 当前缺口（按优先级）

- P0：缺少一键可复现 pipeline（目前 `package.json` 只有 `dsl:build/validate/impact/diff/tailwind`，没有把语义推断与代码生成纳入统一入口）。
- P0：缺少 CI 质量闸门（仓库未发现 `.github/workflows/*`）。
- P1：Schema 校验未与契约文件真正打通（`scripts/dsl/validate.mjs` 主要是规则校验，未消费 `dsl-contract/v1/schema.json`）。
- P1：缺少最小自动化测试（仓库未发现测试文件）。
- P2：文档第十二章“置信度驱动递归”尚未工程化（`confidence < 阈值` 自动加深分析未形成配置与策略落点）。

## 补齐步骤（最小可执行）

1. 统一流水线入口
  - 在 [package.json](/Users/tenghuan/programming/vjshi/code/test-d2c/package.json) 增加：
    - `dsl:semantic` -> `node scripts/semantic-inference.cjs`
    - `dsl:codegen` -> 调用现有或新增生成脚本（输出到 `app/generated`）
    - `dsl:pipeline` -> 串联 `dsl:build -> dsl:validate -> dsl:semantic -> dsl:codegen`
  - 目标：把“阶段1~6”变成可重复执行命令，而不是分散手动执行。
2. 增加 CI 最小闸门
  - 新增 `[.github/workflows/dsl.yml](/Users/tenghuan/programming/vjshi/code/test-d2c/.github/workflows/dsl.yml)`。
  - 首版仅跑：`npm ci && npm run dsl:build && npm run dsl:validate && npm run dsl:semantic`。
  - 目标：每次改动都能自动发现 DSL 结构回归。
3. 打通契约 schema 校验
  - 在 [scripts/dsl/validate.mjs](/Users/tenghuan/programming/vjshi/code/test-d2c/scripts/dsl/validate.mjs) 接入 `Ajv`，消费 [dsl-contract/v1/schema.json](/Users/tenghuan/programming/vjshi/code/test-d2c/dsl-contract/v1/schema.json)。
  - 保留你已有的业务规则校验（ID 唯一性、ref 完整性、依赖环）。
  - 目标：结构合法性（schema）+ 业务一致性（规则）双重校验。
4. 补 3 个最小测试用例
  - 新增测试覆盖：
    - `validate`：schema 失败样例可被拦截
    - `impact`：依赖反查结果稳定
    - `to-tailwind`：核心映射规则不回退
  - 目标：先建立“能防回归”的最小测试网。
5. 增强置信度策略（可后置）
  - 在 [scripts/semantic-inference.cjs](/Users/tenghuan/programming/vjshi/code/test-d2c/scripts/semantic-inference.cjs) 增加阈值配置（如 `CONFIDENCE_THRESHOLD=0.6`）。
  - 当节点推断置信度低于阈值时，触发更深 DFS 或升级到 AI 推断分支。
  - 目标：把文档第十二章建议落成可开关、可调参能力。

## 建议执行顺序

- 第1周：步骤1+2（先保证“可跑、可复现、可拦截”）。
- 第2周：步骤3+4（补校验和测试，降低后续迭代风险）。
- 第3周：步骤5（优化准确率与稳定性）。
